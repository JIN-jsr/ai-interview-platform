# 后端开发演示回答模板

> 以下内容为演示用虚拟材料，不包含真实个人信息。

## MySQL 索引与慢查询

我会先通过慢查询日志和 `EXPLAIN` 看执行计划，重点关注 type、key、rows、Extra 等字段。如果发现没有命中合适索引，会结合 where 条件、排序字段和数据分布设计联合索引，并遵循最左前缀原则。优化后还会观察 QPS、响应时间和数据库 CPU，避免只看单条 SQL。

## Redis 缓存一致性

常见方案是先更新数据库，再删除缓存。为了降低并发场景下的脏数据风险，可以结合延迟双删、消息队列重试和缓存过期时间。对核心数据还会增加监控和降级策略，保证缓存异常时系统仍能读数据库兜底。

## 接口性能优化

我通常会从接口耗时拆解入手，区分网络、数据库、缓存、外部服务和业务计算耗时。优化方式包括减少无效查询、增加缓存、批量化请求、异步处理非核心流程，并通过压测验证优化效果。

## 项目表达

在项目中我更倾向于按“背景、问题、方案、结果”描述。例如某接口高峰期响应慢，我负责定位瓶颈，发现主要耗时来自重复查询和缺少缓存，最终通过索引优化、缓存热点数据和异步写日志把平均响应时间明显降低。
