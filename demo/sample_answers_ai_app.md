# AI 应用开发示例回答

这些回答可以在演示时复制到聊天输入框中。根据实际问题选择相近回答即可。

## 回答 1：RAG 流程

RAG 的核心流程是先把知识文档切分成 chunk，然后用 Embedding 模型转成向量并写入向量数据库，比如 Chroma。用户提问后先做语义检索，召回相关片段，必要时用 Rerank 做二次排序，再把高相关内容和问题一起放进 Prompt，让 LLM 基于检索内容回答。这样可以减少模型幻觉，也方便给答案提供依据。

## 回答 2：Embedding 和向量数据库

Embedding 是把文本映射成向量表示，使语义相近的内容在向量空间中距离更近。向量数据库负责存储这些向量，并根据余弦相似度或其他距离指标做语义检索。在我的项目里，简历、知识点和问题都可以通过关键词或向量方式匹配，从而让面试问题更贴合目标岗位。

## 回答 3：Prompt Engineering

我会把 Prompt 分成角色、任务、输入信息、输出格式和约束条件几部分。比如面试官生成问题时，要提供候选人画像、RAG 知识点、历史回答和要求返回的 JSON 字段。同时要求模型不要脱离知识库自由发挥，并保留备用机制，避免 LLM 失败影响完整流程。

## 回答 4：Function Calling 和 Agent

Function Calling 更适合让模型调用明确工具，比如检索知识库、保存记录或生成报告。Agent 则更强调多步规划和工具组合，例如先分析简历，再检索知识点，然后决定追问方向。实际工程里我会限制工具范围、校验参数 schema，并记录每次工具调用结果，避免不可控行为。

## 回答 5：Token 管理

Token 管理主要包括控制上下文长度、减少无关历史、对文档分块、保留关键摘要，以及限制返回格式。比如连续面试时不能把所有历史原样塞给模型，可以只传最近几轮对话、候选人画像和当前 RAG 条目。这样既能降低成本，也能减少超过上下文窗口导致的失败。

## 回答 6：项目难点

这个项目的难点是让 LLM 生成的问题既自然又有依据。我采用 RAG 先决定要问的知识点，再让 LLM 根据画像和历史回答组织语言。如果 LLM 超时、返回格式错误或生成了重复知识点，系统会切换到备用出题机制，并记录生成方式和切换原因，保证演示时流程稳定。
