# AI 模拟面试与能力提升平台项目设计文档

## 1. 项目背景与目标

计算机相关专业学生在求职前通常需要准备技术基础、项目经历表达和岗位匹配说明，但真实面试机会有限，且学生很难获得系统化反馈。本项目开发一款 AI 模拟面试与能力提升平台，围绕“简历驱动的对话式模拟面试”展开，帮助学生完成从简历解析、面试训练到评分反馈的闭环。

系统目标包括：支持简历输入或上传，自动提取教育背景、技术栈和项目经历；根据简历生成用户画像和面试重点；结合 RAG 知识库进行基础知识考察；围绕项目经历进行连续追问；面试结束后按照五个维度生成评分报告和提升建议。

## 2. 用户流程

用户首先上传 TXT、PDF、DOCX 简历或直接粘贴简历文本。系统解析简历后生成结构化 JSON、用户画像、面试重点和 RAG 推荐问题。随后用户进入模拟面试页面，与 AI 面试官进行连续文字对话。系统会结合简历项目经历、RAG 知识库和用户回答内容进行追问。最后系统根据面试记录生成五维度评分报告。

整体流程为：简历输入/解析 → 用户画像与面试重点 → RAG 知识库检索 → 连续文字面试 → 项目深挖与上下文追问 → 回答分析 → 五维度评分反馈。

## 3. 系统架构

系统采用 Streamlit 快速构建交互界面，Python 负责后端逻辑和数据处理。核心模块位于 `src/` 目录，知识库位于 `data/knowledge_base.json`，文档和测试材料位于 `docs/` 目录。

主要模块包括：

- `resume_file_loader.py`：读取 TXT、PDF、DOCX 简历文本。
- `resume_parser.py`：调用 LLM 或本地规则解析简历。
- `profile_generator.py`：根据结构化简历生成用户画像和面试重点。
- `rag_retriever.py`：从知识库中检索相关基础知识问题。
- `interviewer.py`：控制面试流程、项目追问和 RAG 问题生成。
- `answer_analyzer.py`：分析用户回答长度、关键词、覆盖点和逻辑结构。
- `evaluator.py`：生成正式五维度评分报告。
- `llm_client.py`：封装 OpenAI-compatible LLM API 调用和连接测试。

## 4. 技术栈

项目使用 Python 作为主要开发语言，Streamlit 构建本地网页交互界面，requests 调用 LLM API，python-dotenv 管理本地环境变量，pdfplumber 读取 PDF 简历，python-docx 读取 DOCX 简历，JSON 用于知识库、解析结果和报告数据存储。

本项目使用 OpenAI-compatible LLM API，可配置阿里云百炼千问模型。推荐最终演示使用 `qwen3.7-max`，若模型不可用可切换至 `qwen3.7-plus` 或 `qwen-plus`。

## 5. 简历解析设计

系统支持两种解析模式。若 `.env` 中配置了有效的 LLM API，系统会调用大模型将简历解析为结构化 JSON，包含个人信息、教育背景、技能、项目、实习、竞赛、目标岗位、潜在弱点和面试重点。若 LLM 未配置或调用失败，系统会自动切换为本地规则解析，保证演示稳定性。

这种设计兼顾了智能性和稳定性：正式演示时可使用 LLM 提高解析质量；网络或 API 异常时仍能通过 fallback 完成最小可运行闭环。

## 6. RAG 知识库设计

知识库文件为 `data/knowledge_base.json`，当前包含 80 条有效知识条目，覆盖 Python、Java、数据结构与算法、数据库与缓存、计算机网络、操作系统、软件工程与 AI 应用等方向。

每条知识包含 id、category、tags、difficulty、question、answer、follow_up 和 source 字段。系统根据用户画像中的技能、目标岗位和项目关键词检索相关知识条目，并将其用于基础知识考察和后续追问。

当前版本采用本地关键词检索，匹配 category、tags、question 和 answer 字段，并进行类别多样化筛选。该方案不依赖额外向量数据库或 embedding API，稳定、可解释，适合比赛演示。

## 7. 连续面试与项目深挖设计

面试流程包括自我介绍、项目经历介绍、项目难点深挖、RAG 基础知识考察、基于回答的追问和岗位匹配综合问题。系统会记录每轮问题和用户回答，并根据上一轮回答决定是否继续追问。

例如，若用户在项目回答中提到 Redis、MySQL、缓存或 API，系统会围绕相关技术进一步追问。如果用户在 RAG 基础题中回答过短或覆盖点不足，系统会根据 follow_up 问题或缺失点继续追问，使面试过程更接近真实技术面试。

## 8. 回答分析与评分设计

每轮回答后，系统会分析回答长度、技术关键词、参考答案覆盖度、逻辑线索和临时评分。评分报告基于整个面试记录生成，包含五个维度：基础知识掌握程度 25%、项目理解深度 25%、回答逻辑性 20%、表达完整性 15%、岗位匹配度 15%。

基础知识掌握程度主要参考 RAG 题目的答案覆盖度和技术点命中情况。项目理解深度关注用户是否说明项目背景、个人职责、技术难点和项目结果。回答逻辑性关注表达结构与连接词。表达完整性关注回答长度与信息量。岗位匹配度关注回答是否体现目标岗位相关技能。

最终报告包含总分、等级、维度分数、评分依据、表现较好的方面、主要问题和后续提升建议，并支持 JSON 和 Markdown 下载。

## 9. 交互界面设计

系统界面分为六个页面：简历输入与解析、RAG 知识库、模拟面试、面试记录与分析、评分报告、项目说明与自检。用户可以按照页面顺序完成完整体验，也可以单独测试 RAG 检索、LLM 连接和项目文件完整性。

## 10. API Key 安全处理

真实 API Key 存放在 `.env` 文件中，项目通过 python-dotenv 读取环境变量。`.gitignore` 已忽略 `.env`，避免密钥上传到 GitHub。仓库中只保留 `.env.example`，用于说明需要配置的字段。

## 11. 测试情况

项目已设计并完成以下测试：简历文本输入测试、TXT/PDF/DOCX 文件读取测试、LLM 连接测试、本地 fallback 测试、RAG 知识库检索测试、连续面试流程测试、上下文追问测试、面试记录 JSON 下载测试、正式评分报告生成测试、报告 Markdown 下载测试和项目自检脚本测试。

## 12. AI 辅助开发说明

本项目开发过程中使用 AI 工具辅助完成代码框架、文档初稿、测试清单和部分实现建议。项目功能设计、模块整合、运行测试和最终提交材料由开发者根据赛题要求进行整理和确认。真实 API Key 未写入代码仓库。

## 13. 总结

本项目完成了从简历输入到评分反馈的完整闭环，覆盖赛题要求的简历理解、文字模拟面试、RAG 基础知识考察、项目经历深挖、评分反馈、本地运行和演示说明。系统具有较好的可运行性、可解释性和扩展性，后续可以进一步加入向量检索、语音输入、岗位题库扩展和更精细的 LLM 评分反馈。
